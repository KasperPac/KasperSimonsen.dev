01 Brief

Executive summary

Forja is an AI-assisted automation engineering platform that compresses the time required to design, specify, and code a Siemens TIA Portal project. It pairs a structured functional-spec authoring environment — the FDS Builder — with a guided code-generation wizard — the Forge Wizard — backed by a continuously-learning pattern library and a catalogue of pre-built, version-controlled function block templates.

Built initially as an internal tool for Pac Technologies engineers, the goal is to allow a single engineer to run multiple projects in parallel — turning the studio's capacity constraint from engineer-hours per project into projects per engineer. A licensing path to external automation firms is a secondary, longer-horizon objective once the tool is proven internally.

The MVP delivers four integrated modules: structured spec authoring, a fifteen-step code-generation wizard, a continuously-learning correction pattern library, and a versioned catalogue of pre-built function block templates. HMI screen generation, third-party robot vendor integration, and external licensing are explicitly out of scope for the MVP — see Appendix B.

02 Brief

Project brief

Background

Pac Technologies delivers Siemens TIA Portal automation projects to industrial customers across Australia and New Zealand. A typical mid-sized project takes a senior engineer four to eight weeks — moving from a customer's functional brief through device specification, code authoring, HMI development, simulation testing, and on-site commissioning.

Most of that time is repetitive scaffolding. The same patterns of conveyor logic, motor sequencing, and alarm handling get written, by hand, project after project.

Modern AI tools can generate plausible PLC code from a prompt, but they make systematic mistakes the moment the project deviates from common patterns: incorrect IO mapping, missed safety interlocks, inconsistent naming, code that compiles but doesn't behave correctly. A general-purpose AI is not enough.

The engineering knowledge embedded in a senior engineer's head — house style, customer conventions, vendor quirks, hard-won corrections — needs a place to live and a way to influence every generation. Forja is that place.

Goals

Business

  • Compress the engineer-hours per project by 3× or more on representative work.
  • Run multiple concurrent projects without losing code quality.
  • Capture institutional knowledge in a form that survives staff turnover.
  • Establish a foundation that can be commercialised externally — without rebuilding from scratch.

Engineering

  • Reduce the cognitive load of switching between projects.
  • Eliminate the most common categories of repeated rework — naming, IO wiring, missed permissives.
  • Produce TIA-importable code that compiles first attempt for at least 80% of generated artefacts.
  • Make the engineer's review the primary act of value, not the typing.

Users & personas

Three personas, in priority order.

Primary user — 01

Senior automation engineer

  • Ten-plus years on TIA Portal, fluent in SCL and LAD.
  • Currently runs one or two concurrent projects; wants to run three or four without dropping quality.
  • Has strong opinions about house style and naming conventions.
  • Sceptical of AI tools that produce plausible but subtly wrong code.
  • Wants the tool to disappear when it's working and surface clearly when it isn't.
  • Will not adopt anything that costs more time than it saves on day one.

Sponsor — 02

Studio principal

  • Sets project pricing and capacity targets.
  • Watches engineer utilisation and project margin.
  • Cares about Forja's effect on throughput and quality, not its internals.
  • Holds the budget for ongoing operational costs.
  • Measures success by projects delivered and customer satisfaction — not by lines of code generated.
Secondary — 03 — post-MVP

Junior automation engineer

  • Two to four years' experience.
  • Currently restricted to simpler projects or works under supervision.
  • Forja could let them deliver projects independently sooner — with the tool encoding the senior's review patterns.
  • Not a primary MVP user. Their needs shape Phase 2 design.
Jobs to be done

What the engineer is trying to accomplish

Primary

When I receive a functional brief from a customer, I want to author a structured functional specification in a form that drives downstream code generation — so I produce executable engineering rather than prose that has to be re-interpreted later.

When I have a structured spec, I want to generate a complete TIA Portal project that compiles on first import — so my time is spent on review and refinement rather than typing boilerplate.

When the AI makes a mistake I have to correct, I want that correction captured and applied to all future generations — so I never make the same fix twice.

Secondary

When I'm running multiple projects, I want each one to remember exactly where I left off — so context-switching costs are minimised.

When I encounter a structurally familiar machine — a conveyor, a pusher, a lift table — I want the tool to use a proven template rather than re-generate from scratch — so output quality is dictated by my best work, not the AI's average.

When I'm uncertain about a Siemens instruction or convention, I want the tool to retrieve the relevant section of the official documentation and use it — so I don't have to break flow to look things up.

06 Scope

Module specifications

Feature definition with acceptance criteria. Implementation detail lives in the technical architecture and per-module engineering notes alongside the codebase.

06.1

FDS Builder — Functional Description Specification authoring

The FDS Builder is the engineering side of Forja. It produces the same functional specification document a senior engineer would write by hand — but as structured, machine-readable data rather than prose.

Phases

  1. Instrument Register Upload — upload an IO spreadsheet (CSV or Excel). The system classifies each tag and stores it as the authoritative tag source for all subsequent phases.
  2. Spec Skeleton Wizard — six-step guided form to define the machine hierarchy: System / Subsystem / Assembly / Device, plus operating states and alarm tiers.
  3. FDS Co-Authoring — per-assembly behavioural specification, conducted as a structured AI interview. Static device states are auto-filled deterministically; sequential states are co-authored with an AI agent.
  4. System Orchestration — subsystem-to-subsystem coordination: ordering, shared permissives, inter-subsystem interlocks.
  5. Structured Spec Editor — document tree navigation, section-by-section approval flow.
  6. DOCX Export — formatted Word output in the Pac Technologies house style.

Acceptance criteria

  • An engineer can produce a complete behavioural specification for a 5-station, 25-device machine in under a working day.
  • The spec is the authoritative source of truth for downstream code generation — no re-interpretation.
  • For structurally identical assemblies, behavioural data can be cloned and tag-remapped automatically.
  • DOCX output is presentable to a customer without manual reformatting.

  • A wizard session can be paused and resumed at any step without state loss.
  • Each step exposes its outputs for engineer review before the next step proceeds.
  • Generation is parameterised by the engineer's chosen design profile — naming, language, conventions.
  • Library-matched artefacts are copied verbatim with no AI involvement; AI generation is a fallback for novel cases.
  • The deterministic logic checker (Step 08) provides actionable, FDS-grounded findings — not generic advice.
06.3

Pattern Librarian — Continuous learning

Every code correction in Forja — whether from compile-fix, manual TIA fix, pipeline review, or engineer chat — is captured as a structured WRONG/CORRECT pattern. Approved patterns are injected into every future generation prompt so the same mistake is never repeated.

Capture sources

  • Forge Wizard rewrite loop (per artefact)
  • Forge Wizard compile-fix cycles — auto-approved on successful compile
  • Manual TIA fix panel
  • Direct engineer teaching via chat
  • Document-extraction teaching dialogs

Classification

Patterns are classified into one of six types — NAMING · IO_MAPPING · STATE_LOGIC · ALARM · SAFETY · TIMING. AI classification is the primary path; a regex-based classifier is the deterministic fallback.

Lifecycle

PENDING  —  APPROVED or REJECTED, with revoke and delete actions. Only approved patterns are injected into future generation prompts.

Acceptance criteria

  • Pattern capture is fire-and-forget — never blocks the main generation flow.
  • Admin review queue defaults to PENDING — engineers see new patterns first.
  • Patterns are scoped to a PLC brand (currently SIEMENS_TIA), allowing future expansion without contamination.
  • An approved pattern visibly affects future generations — verifiable by re-running a previously failed generation.

06.5

Reference Library — Indexed authoritative documentation

Engineers upload Siemens manuals, instruction references, style guides, and customer conventions. Forja splits each document into searchable sections with topic tags. At generation time, a two-pass AI retrieval pulls the most relevant sections into the prompt as authoritative context.

Two-pass retrieval

  1. The user's request — or generated code, or compile errors — is analysed by AI to extract relevant topic keywords.
  2. A Postgres full-text + tag-overlap search returns the matching reference sections.
  3. The sections are formatted and injected into the system prompt for the generation step.

Acceptance criteria

  • A 500-page manual can be uploaded and indexed in under fifteen minutes.
  • Engineers can browse, search, and tag sections through a UI.
  • Reference results are filtered by programming language (LAD / SCL / general) — an SCL generation doesn't get LAD documentation.
  • All four AI generation paths consume the reference library uniformly.
07 Architecture

Cross-module integration

Forja's modules are loosely coupled and clearly bounded. The integration map below documents how data flows between them.

The handoff that matters most

The handoff from FDS Builder to Forge Wizard is the single most important integration in the system. It's implemented as a Postgres function — _build_contract_snapshot() — that returns a complete spec contract as a JSON snapshot.

Forge Wizard reads from the snapshot, never from the FDS Builder's internal tables. This boundary protects both modules from one another's internal evolution.

The Reference Library and Pattern Librarian are read-only services consumed by all generation paths uniformly through a single shared service layer. This is deliberately designed to avoid the trap of "four generation paths each with their own context-building logic" — a known anti-pattern in AI-driven systems.

The Assembly FB Library is consumed at two points — at FDS Builder time (template selection per assembly during the hierarchy phase) and at Forge Wizard time (artefact emission during assembly FB generation). Both consumption points read from the same fb_templates table.

System diagram

Architectural principles

Non-negotiable in MVP design and review.

  1. Module boundaries are honoured. The five modules communicate through a small set of well-typed contracts. No module reaches across into another module's tables.
  2. Deterministic over generative wherever possible. AI is used where structure cannot be derived; everything else is deterministic and auditable.
  3. Generation context is centralised. All AI generation paths consume a single shared service that assembles patterns, knowledge, reference sections, design profiles, and platform rules.
  4. Spec contracts are immutable snapshots. The handoff between modules is a versioned snapshot, not a live reference. This protects each module from the others' internal evolution.
  5. Every artefact is auditable. Generated outputs carry their lineage — which prompt produced them, which patterns were active, which reference sections were retrieved.
  6. Engineer review is the value gate. Forja generates; engineers approve. The tool never deploys directly to customer systems.

Two design choices worth noting

  • JSONB is used liberally for structured-but-evolving payloads — interface contracts, sequence steps, behavioural state tables. Each JSONB column has a documented schema and a TypeScript type derived from a Zod schema in the frontend.
  • Spec contracts are assembled by a Postgres function — _build_contract_snapshot — rather than reconstructed in application code. This guarantees the FDS Builder's view of a spec and the Forge Wizard's view are identical.

12 Delivery

Hosting & deployment plan

The hosting plan evolves in three stages.

Stage 01 — Months 0–6

Local server

Forja is hosted on a single workstation inside the studio LAN. Supabase runs via Docker locally for development and against a hosted Supabase project for shared state. Minimises operational complexity while the tool is in active development.

Stage 02 — Months 6–9

Hosted Supabase, local frontend

Supabase moves to its hosted offering. The frontend continues to run from the engineer's machine via npm run dev for the initial pilot period. Validates the cloud database under real use before exposing the frontend.

Stage 03 — Months 9+

Vercel-hosted frontend

The frontend deploys to Vercel with environment-based access controls. The TIA Bridge continues to run locally on each engineer's Windows machine — this cannot be hosted, by design.

External-customer hosting is not addressed in this document and would require a separate architecture review — multi-tenancy, customer data isolation, and pricing model.

13 Risk

Security, privacy & IP protection

Customer project data

Customer functional specs may contain commercially sensitive information about the customer's process, equipment, and IP. Treatment of this data is non-negotiable.

  • All customer project data is stored in Supabase under row-level security policies scoped to the originating engineer.
  • Customer data is never shared with the AI provider beyond the immediate generation context, and is never used to train the AI provider's models. Anthropic's API terms support this — explicit verification at sign-up.
  • Customer files (DOCX, PDF, Excel) are stored in Supabase Storage with signed URLs and a default expiry.
  • Engineers can export and delete a project's full data on request.

Pac Technologies IP

The function block library, reference documents, correction patterns, design profiles, and prompt sections collectively constitute Pac Technologies' engineering IP.

  • These artefacts live in Supabase, backed up daily.
  • The catalogue is versioned — destructive changes are recoverable.
  • A planned Phase 2 capability is "library export" — a portable bundle of the studio's IP that can be migrated, backed up offsite, or, eventually, licensed to external customers.

Authentication

Supabase email-and-password authentication for MVP. SSO via the studio's existing identity provider considered for a future phase if user count grows.

14 Delivery

Testing & validation strategy

Three layers of testing, in order of priority.

01 — Authoritative

Compile validation against TIA Portal

The most authoritative test for generated code is whether TIA Portal accepts it. Every Forge Wizard run ends with an automated compile pass. Compile failures surface and route into the compile-fix loop.

02 — Coverage

Deterministic output tests

The deterministic generators — process compiler, IO linking, OB1 scaffold, type-conversion FCs — have unit tests in Vitest covering representative inputs. They run on every commit. Target coverage at MVP: 70% line coverage on the deterministic libraries, 0% on the AI-driven paths.

03 — Specification

Structured logic checks

The Step 08 logic check — generated FB code validated against the FDS spec across ten categories — is itself a deterministic test layer. It catches the highest-impact category of error: AI generates code that compiles but doesn't behave per spec.

Not tested at MVP

  • AI-generated outputs are not unit-tested. The AI is non-deterministic and asserting on its outputs costs more than it's worth at MVP scale. Engineer review is the gate.
  • HMI generation — deferred to a future phase.
  • Multi-engineer concurrent editing — the MVP user count makes this low-priority.

18 Risk

Assumptions & open questions

Assumptions

  • The studio's two to three engineers are willing to use Forja on real projects from the pilot phase onward.
  • The TIA Portal Openness API remains stable across V17–V20 for the duration of MVP development.
  • Anthropic's API pricing remains within an order of magnitude of current rates.
  • No regulatory regime — functional safety certification, customer-imposed code review process — blocks AI-generated code from reaching TIA Portal in customer projects.
  • Pac Technologies has the right to use historical projects as training inputs to the pattern library and reference library.

Open questions

Require resolution before, or during, Phase 00.

  1. Pilot project selection. Which two real customer projects will be delivered through Forja during the pilot phase? Choice affects which assembly templates are prioritised in V1 catalogue.
  2. Code review policy for generated artefacts. Does AI-generated code require a second engineer's sign-off before TIA upload, or is the generating engineer's review sufficient?
  3. Library contribution policy. Who can author new function block templates and reference library entries — any engineer, or only the principal? Affects governance design.
  4. Pattern approval policy. Are patterns auto-approved by the engineer who generated them, or do they queue for principal review? Trade-off between learning velocity and quality control.
  5. Backup and disaster recovery scope. What's the studio's tolerance for data loss? Affects backup frequency and retention policy.
  6. Failure-mode policy. When Forja cannot handle a project segment — a third-party vendor controller — what's the documented fallback?
  7. External licensing direction. What's the target window for considering external licensing — twelve months, twenty-four? Affects whether multi-tenant architecture should influence MVP design choices.
  8. Customer disclosure. Do customers need to know AI was used in producing their project code? Affects contracts, liability, and marketing.

Glossary

AssemblyA coordinated group of devices working together — a conveyor, a lift table, a pusher.
DeviceA single physical thing with IO signals — a motor, a limit switch, a solenoid.
DOLDirect-on-line motor — uncontrolled start, contactor only.
FBFunction Block (Siemens). Reusable block of code with internal state, instantiated as instance DBs.
FCFunction (Siemens). Stateless block of code.
FDSFunctional Description Specification. The structured behavioural document produced by the FDS Builder.
HMIHuman-Machine Interface — the operator screens.
LADLadder Diagram. One of the IEC 61131-3 PLC programming languages.
OBOrganization Block (Siemens). Top-level entry point.
PLCProgrammable Logic Controller.
PLCSimSiemens' PLC simulator. Used for offline testing without physical hardware.
SCLStructured Control Language. Pascal-like PLC language; one of the IEC 61131-3 standards.
SubsystemA functional station within a machine — infeed conveyor station, hydraulic lift station.
TIA PortalSiemens' integrated engineering environment for Siemens controllers and HMIs.
TIA OpennessSiemens' API for programmatic interaction with TIA Portal projects.
UDTUser-Defined Type. A struct-like typed record in TIA Portal.
VSDVariable Speed Drive — controlled motor speed via drive electronics.
